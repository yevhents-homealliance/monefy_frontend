Відповіді на питання: 

1. Як я структурував frontend і backend?
Backend (Laravel)

Я розділив відповідальності таким чином:
    - Controllers – приймають HTTP-запити та повертають API-відповіді
    - Form Request – відповідають за валідацію
    - Service layer (InvoiceService) – містить бізнес-логіку (розрахунок gross_amount, зміна статусу тощо)
    - Models (Eloquent) – робота з БД
    - API Resources – нормалізація JSON-відповідей

Основна ідея:
Контролер не містить бізнес-логіку, лише координує запит → сервіс → відповідь.

Це дозволяє:
    - легко тестувати бізнес-логіку
    - не перевантажувати контролери
    - масштабувати функціональність

Frontend (Nuxt)

Frontend розділений так:

    - pages/ – сторінки (index, create, edit)
    - components/ – перевикористовувані UI-компоненти
    - composables/useInvoices.ts – робота з API

Основна логіка роботи з API винесена в composable, щоб:

    - не дублювати код
    - зберігати чистоту сторінок
    - централізувати запити

2. Які компроміси я зробив і чому?

    - Не реалізував повноцінну авторизацію (JWT / Sanctum), бо це виходить за рамки тестового.
    - Не додав unit/feature тести через обмеження часу.
    - Мінімізував складність UI (без глобального state management), бо для невеликого CRUD це надлишково.
    - Не реалізовував кешування на backend (для тестового немає реальної потреби).

Ці рішення дозволили зосередитись на:
    - правильній архітектурі
    - бізнес-логіці
    - чистоті коду

3. Що б я покращив у production-версії?
Backend:

    - Додав би unit та feature тести
    - Додав би DTO для ще чіткішого відділення даних
    - Реалізував би авторизацію (Laravel Sanctum)
    - Додав би логування та централізовану обробку помилок
    - Додав би pagination + фільтрацію + сортування

Frontend:

    - Додав би глобальний error handling
    - Покращив би UX при завантаженні (skeleton loaders)
    - Додав би optimistic updates
    - Додав би form-level error handling
    - Реалізував би debounce для пошуку

4. Які UX edge cases я врахував?

    - Валідація обовʼязкових полів
    - Заборона збереження, якщо net_amount > gross_amount
    - Блокування кнопки під час сабміту (щоб уникнути дублювання запиту)
    - Обробка помилки 500/422 з бекенду
    - Відображення стану invoice (pending / approved)
    - Обробка ситуації, коли API повертає порожній список
